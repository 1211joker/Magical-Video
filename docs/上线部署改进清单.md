## 🔴 P0 — 安全短板（5 项）

### P0-1: CORS 配置全开放

**位置**：`backend/main.py` 第 28 行

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],       # ← 任何网站都能调用你的 API
    allow_credentials=True,     # ← 允许携带凭据
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**风险**：任何恶意网站都可以从前端脚本调用你的后端 API，包括消耗 DeepSeek 余额进行 AI 分析、发起大量下载请求消耗带宽和服务器存储。

**建议方案**：
- 通过环境变量 `ALLOWED_ORIGINS` 配置白名单
- 生产环境只允许部署域名，如 `https://your-domain.com`

**✅ 已实施（2026-06-30）**：

| 文件 | 改动内容 |
|------|----------|
| `backend/config.py` | 新增 `ALLOWED_ORIGINS = os.getenv("ALLOWED_ORIGINS", "*")` |
| `backend/main.py` | 导入 `ALLOWED_ORIGINS`，CORS 中间件改为 `origins = ALLOWED_ORIGINS.split(",") if ALLOWED_ORIGINS and ALLOWED_ORIGINS != "*" else ["*"]` |
| `backend/.env.example` | 新增 `ALLOWED_ORIGINS=*` 配置项及注释说明 |

行为：未设置环境变量或设为 `*` 时保持开发模式（允许所有来源）；生产环境在 `.env` 中设置 `ALLOWED_ORIGINS=https://your-domain.com` 即可启动白名单。

### P0-2: 限流机制未实际生效

**位置**：
- 定义：`backend/config.py` 第 14-15 行
- 使用：**无**（全代码库搜索确认未在任何地方引用）

```python
# config.py — 定义了但从未使用
AI_RATE_LIMIT_PER_MIN = 10
MAX_CONCURRENT_TASKS = 2
```

**验证方法**：
- `grep -rn "AI_RATE_LIMIT_PER_MIN\|MAX_CONCURRENT_TASKS" backend/` 仅在 config.py 中出现
- `grep -rn "slowapi\|rate_limit\|RateLimiter\|limiter\|throttle" backend/` 无结果

**风险**：任何 IP 可以无限次调用 `/api/analyze-stream`（每次消耗 DeepSeek token 费用）、`/api/download`（每次消耗带宽和磁盘 I/O）。如果 DeepSeek API Key 泄露或用户滥用，数小时内可能产生巨额费用。

**建议方案**：
- 引入 `slowapi`（FastAPI 原生兼容，基于 `limits` 库）
- 对 `/api/analyze-stream` 和 `/api/ask` 限制 10次/分钟/IP
- 对 `/api/download` 限制 5次/分钟/IP

**✅ 已实施（2026-06-30）**：

| 文件 | 改动内容 |
|------|----------|
| `backend/requirements.txt` | 新增 `slowapi>=0.1.9` + `limits>=5.0` 依赖 |
| `backend/limiter.py` | **新建** — 共享 `Limiter(key_func=get_remote_address)` 实例，供所有路由模块导入，避免循环引用 |
| `backend/main.py` | `app.state.limiter = limiter` + 注册 `RateLimitExceeded` 异常处理器 |
| `backend/routers/analyze.py` | 导入 `limiter`；`/parse` 加 `@limiter.limit("20/minute")`；`/thumbnail` 加 `@limiter.limit("30/minute")`；`/analyze-stream` 加 `@limiter.limit("5/minute")`；各函数签名新增 `request: Request` 参数（slowapi 用其获取调用方 IP） |
| `backend/routers/qa.py` | `/ask` 加 `@limiter.limit("10/minute")` |
| `backend/routers/download.py` | `/download` 加 `@limiter.limit("3/minute")` |

实际限流值：

| 端点 | 限制 | 理由 |
|------|------|------|
| `/api/parse` | 20次/分 | 轻量元数据查询，yt-dlp 子进程开销 |
| `/api/thumbnail` | 30次/分 | 图片代理，带宽开销 |
| `/api/analyze-stream` | **5次/分** | DeepSeek API **直接计费**，最严格 |
| `/api/ask` | 10次/分 | DeepSeek API 调用，较 analyze 轻量 |
| `/api/download` | **3次/分** | 视频下载，**带宽 + 磁盘**重消耗 |
| `/api/health` | 不限 | 健康检查无资源消耗 |

**🐛 Bug 修复（2026-06-30，测试发现）**：

`/api/download` 限流未生效 — slowapi 通过参数**名称**（非类型）查找 `Request` 对象。原代码 `request: DownloadRequest` + `req: Request` 导致 slowapi 拿到 Pydantic model 而非真实 Request，限流静默失效。

修复：`routers/download.py` 中 `request: DownloadRequest` → `body: DownloadRequest`，`req: Request` → `request: Request`。修复后验证：3 次正常请求后第 4 次正确返回 429。

已通过全端点限流测试：所有 5 个受限端点均在第 N+1 次请求时正确触发 429。

### P0-3: 无任何用户认证/授权机制

**位置**：全局 — `backend/main.py`、所有路由文件

**现状**：所有 7 个 API 端点完全公开访问：

| 端点 | 方法 | 资源消耗 |
|------|------|----------|
| `/api/health` | GET | 无 |
| `/api/parse` | POST | yt-dlp 子进程（CPU + 网络） |
| `/api/thumbnail` | GET | 代理下载图片（带宽） |
| `/api/analyze-stream` | POST | DeepSeek API token（**直接计费**） |
| `/api/download` | POST | 视频下载（**带宽 + 磁盘**） |
| `/api/ask` | POST | DeepSeek API token（**直接计费**） |

**风险**：DeepSeek API 按 token 计费，视频下载消耗大量带宽和临时磁盘空间。没有任何认证保护的情况下，任何人发现这个服务都可以无限制使用付费资源。

**建议方案**（按实现复杂度排序）：
1. **最简方案**：nginx 反向代理层配置 HTTP Basic Auth（`ngx_http_auth_basic_module`），5 分钟可完成
2. **中等方案**：服务端 API Key 机制 — 用户在请求头中携带 `Authorization: Bearer <api-key>`，后端中间件校验
3. **完整方案**：JWT + 用户注册/登录系统 + 用量配额

**✅ 已实施 — nginx Basic Auth（2026-06-30）**：

实施步骤（Ubuntu + 宝塔面板）：

**第 1 步：SSH 登录服务器，创建密码文件**
```bash
sudo apt install -y apache2-utils
sudo mkdir -p /www/server/pass
sudo htpasswd -c /www/server/pass/magical-video.pass your-username
# 输入两次密码
```

**第 2 步：宝塔面板配置 nginx**
宝塔面板 → 网站 → 站点 → 配置文件 → 在 `location / { }` 块内加入：
```nginx
auth_basic "Magical Video";
auth_basic_user_file /www/server/pass/magical-video.pass;
```
保存后宝塔自动重载 nginx。

**效果**：任何人访问网站或 API 时，浏览器弹出原生的用户名密码对话框。不知道密码 = 完全无法访问页面和 API，DeepSeek Key 和下载资源获得第一层保护。

**⚠️ 注意事项**：
- Basic Auth 的用户名密码通过 HTTP 头明文传输，**必须配合 HTTPS** 使用，否则密码可被中间人嗅探
- 如需添加更多用户：`sudo htpasswd /www/server/pass/magical-video.pass another-user`（去掉 `-c` 参数，`-c` 只在首次创建时使用）
- 与 P0-1（CORS）和 P0-2（限流）形成三层防护：**认证 → 限流 → 白名单**

### P0-4: yt-dlp 调用禁用 SSL 证书验证

**位置**：`backend/services/ytdlp_service.py` 第 106、280、362 行（共 3 处）

```python
cmd = [
    sys.executable, "-m", "yt_dlp",
    "--no-check-certificates",   # ← 跳过 HTTPS 证书验证
    ...
]
```

**风险**：禁用证书验证意味着 yt-dlp 发出的所有 HTTPS 请求都不会验证服务器的 SSL 证书，可能被中间人攻击（MITM），篡改返回的视频元数据或植入恶意内容。

**✅ 已实施 — 环境变量控制（2026-06-30）**：

| 文件 | 改动内容 |
|------|----------|
| `backend/config.py` | 新增 `YTDLP_NO_CHECK_CERTIFICATES = os.getenv("YTDLP_NO_CHECK_CERTIFICATES", "false").lower() == "true"` |
| `backend/services/ytdlp_service.py` | 导入 `YTDLP_NO_CHECK_CERTIFICATES`；3 处 `--no-check-certificates` 从硬编码改为 `if YTDLP_NO_CHECK_CERTIFICATES: cmd.insert(...)` 条件追加 |
| `backend/.env.example` | 新增 `YTDLP_NO_CHECK_CERTIFICATES` 注释说明 |

行为：
- **默认安全**：不设置环境变量时，yt-dlp 正常验证 SSL 证书
- **临时回退**：设置 `YTDLP_NO_CHECK_CERTIFICATES=true` 后恢复跳过证书验证（仅用于国内网络劫持环境）

### P0-5: 缺少请求体大小限制

**位置**：全局 — 后端无任何请求体大小上限

**验证方法**：`grep -rn "max_body_size\|max_request_size" backend/` 无结果

**风险**：`/api/parse`、`/api/analyze-stream`、`/api/download` 都接受 `cookies` 文本字段（可以很大），`/api/ask` 接受完整的字幕文本（可能 20,000+ 字符）。恶意用户可以发送超大请求体导致服务内存耗尽（OOM）。

**建议方案**：
- FastAPI 层面：通过 Starlette 的 `MaximumContentLength` 中间件设置上限（如 5MB）
- 反向代理层面：nginx `client_max_body_size 10m`
- 对 cookies 文本字段在 Pydantic 模型中添加 `max_length` 约束

## 🟠 P1 — 可靠性短板（5 项）

### P1-1: SSE 长连接无心跳保活

**位置**：`backend/routers/analyze.py` 第 100-153 行 `event_generator()`

**现状**：AI 分析（DeepSeek API 调用）耗时 60-120 秒，期间 SSE 连接无任何数据发送。如果部署在 nginx/负载均衡器后面，默认的 `proxy_read_timeout`（通常 60s）会导致连接被中间代理关闭，前端收到 504。

**验证方法**：`grep -rn "heartbeat\|keepalive\|: keep\|ping" backend/routers/analyze.py` 无结果

**建议方案**：
- 在 `event_generator` 中添加 `asyncio.create_task` 定时心跳 — 每 15 秒发送 `:keepalive\n\n`（SSE 注释格式，前端忽略但中间代理识别为活跃连接）
- 部署时 nginx 配置 `proxy_read_timeout 300s`（已在 `vite.config.js` 中为开发环境配置了 `timeout: 300000`，生产环境需要同步配置）

### P1-2: DeepSeek API 调用无重试机制

**位置**：
- `backend/services/deepseek_service.py` 第 125-137 行
- `backend/routers/qa.py` 第 137-152 行

**现状**：调用 DeepSeek API 遇到 429（限流）、5xx（服务端临时故障）、网络超时等可恢复错误时，直接抛出异常返回给用户，无任何重试。

**风险**：DeepSeek API 偶尔的波动导致用户体验断裂，用户需要手动重试。

**✅ 已实施 — tenacity 指数退避重试（2026-06-30）**：

| 文件 | 改动内容 |
|------|----------|
| `backend/requirements.txt` | 新增 `tenacity>=8.0` 依赖 |
| `backend/services/deepseek_client.py` | **新建** — 共享 DeepSeek HTTP 客户端。`RetryableError` 异常类 + `call_deepseek_api()` 函数，带 `@retry(stop=3, wait=exponential(2s→4s→8s))` 装饰器，仅对 429/5xx/超时可重试 |
| `backend/services/deepseek_service.py` | 移除直接 `httpx.AsyncClient` 调用，改用 `call_deepseek_api()`；移除不再需要的 `DEEPSEEK_API_URL` 导入 |
| `backend/routers/qa.py` | 同上，改用 `call_deepseek_api()`；新增 `RetryableError` 异常处理 |

重试策略：
- **重试次数**：最多 3 次
- **退避间隔**：2 秒 → 4 秒 → 8 秒（指数增长）
- **可重试**：429（限流）/ 500 / 502 / 503 / 504 / 网络超时
- **不重试**：401（Key 无效）/ 402（余额不足）/ 4xx 客户端错误
- 仅对 429 / 5xx / 网络超时可重试，4xx 错误不重试

### P1-3: httpx 客户端每次请求新建，无连接池复用

**位置**（3 处）：
- `backend/services/deepseek_service.py` 第 125 行：`async with httpx.AsyncClient(timeout=..., proxy=None) as client:`
- `backend/routers/analyze.py` 第 68 行：`async with httpx.AsyncClient(timeout=15, ...) as client:`
- `backend/routers/qa.py` 第 138 行：`async with httpx.AsyncClient(timeout=60) as client:`

**现状**：每个请求都创建新的 `AsyncClient` 实例，用完即销毁，无法利用 HTTP keep-alive 连接复用。每次 TCP 握手 + TLS 协商增加 50-200ms 延迟。

**建议方案**：
- 在 FastAPI `startup` 事件中创建全局 `httpx.AsyncClient` 实例
- 配置连接池参数：`limits=httpx.Limits(max_keepalive_connections=20, max_connections=50)`
- 在 `shutdown` 事件中优雅关闭

---

### P1-4: 下载流程存在中间故障点

**位置**：`backend/services/ytdlp_service.py` 第 307-415 行 `download_video()`

**现状**：下载流程为 yt-dlp 完整下载到临时目录 → 文件落盘 → `FileResponse` 发送。对于大视频文件：
- 用户需等待双倍时间（下载到服务器 + 传输到浏览器）
- 磁盘空间被占用（文件越大，占用越多）
- 如果服务器磁盘空间不足，下载会直接失败

**建议方案**：
- 保留当前方案作为 fallback
- 对于不需要 ffmpeg 合并的场景（如 format 18 的 YouTube 360p），使用 yt-dlp `-o -` 输出到 stdout → `StreamingResponse` 流式传输
- 生产环境配置 nginx `proxy_buffering off` 以支持真正的流式代理

### P1-5: 无优雅关闭（Graceful Shutdown）机制

**位置**：`backend/main.py` — 未注册任何 shutdown 事件处理器

**验证方法**：`grep -rn "on_event\|shutdown\|lifespan" backend/main.py` 无结果

**现状**：服务器重启 / 停止时：
- 正在进行的 yt-dlp 子进程变成孤儿进程
- 临时下载文件残留（`/tmp/ytdlp_dl_*`、`/tmp/bili_subs_*`）
- 正在进行的 DeepSeek API 调用被直接中断
- cookies 临时文件可能未被清理

**建议方案**：
- 在 `@app.on_event("shutdown")` 中：
  - 遍历活跃的 asyncio 子进程并发送 SIGTERM
  - 清理所有临时目录
  - 等待进行中的 HTTP 请求完成（设置最大等待时间）

### P3-2: 日志系统不适用于生产环境

**位置**：`backend/main.py` 第 9-13 行

```python
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(name)s] %(levelname)s: %(message)s',
    datefmt='%H:%M:%S'
)
```

**现状**：纯文本格式输出到 stdout，无结构化字段（request_id、user_id、duration_ms），不适合日志聚合系统（ELK / Loki / Datadog）。

**建议方案**：引入 `structlog` 或自定义 `logging.Formatter` 输出 JSON 格式日志，包含 `timestamp`、`level`、`logger`、`message`、`request_id`、`duration_ms` 等字段。

### P3-3: 无错误追踪/监控系统

**位置**：全局

**验证方法**：`grep -rn "sentry\|Sentry\|prometheus\|metrics" backend/` 无结果

**现状**：当后端发生异常时，只有控制台日志，开发者无法主动感知线上错误。

**建议方案**：集成 Sentry SDK（`sentry-sdk[fastapi]`），成本极低（一行导入 + 一行初始化），即可捕获所有未处理异常并发送告警。

### P3-4: 前端大文件下载使用内存 Blob

**位置**：`frontend/src/api/index.js` 第 166 行

```javascript
const blob = await response.blob()  // ← 将整个视频文件读入浏览器内存
```

**现状**：对于数百 MB 的视频文件，这会导致浏览器内存耗尽、页面卡死。

**建议方案**：对于返回 `Content-Disposition: attachment` 的响应，不使用 fetch + Blob 路径，改用直接导航或 `<a href>` + `download` 属性触发浏览器原生下载管理器（不经过 JS 内存）。

---

### P3-5: 缺少安全 HTTP 头配置

**位置**：`frontend/index.html` 和生产部署配置

**验证方法**：`grep -rn "Content-Security-Policy\|X-Content-Type\|X-Frame\|Strict-Transport" frontend/` 无结果

**现状**：无 CSP、无 HSTS、无 X-Frame-Options、无 X-Content-Type-Options。这些是 OWASP 推荐的基础安全头。

**建议方案**：在 nginx 层面统一配置：

```nginx
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self'";
add_header X-Content-Type-Options "nosniff";
add_header X-Frame-Options "DENY";
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```
