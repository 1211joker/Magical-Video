# AI 万能视频下载总结器 — 项目文档

> 基于 yt-dlp + DeepSeek AI 的 Web 应用，让用户在下载视频前先了解核心内容，判断值不值得看。

---

## 文档总览

本项目包含 **5 份 Markdown 文档**，按产品开发流程顺序排列：

```
discovery-plan.md         ← 产品发现：验证想法是否靠谱
    │
    ▼
competitor-analysis.md    ← 竞品分析：了解市场格局
    │
    ▼
prd.md                    ← 产品需求文档：定义做什么
    │

    │
    └── user-stories-dev.md ← 开发任务拆解：指导怎么做
```

## 文档详情

### 1. `discovery-plan.md` — 产品发现计划

**作用**：从零开始验证"这个产品值不值得做"

**内容**：
- 从 PM/Designer/Engineer 三视角生成了 15 个创意，筛选出 4 个核心创意
- 记录了**真实环境技术实验**结果（YouTube ✅ / B站 ✅ / 抖音 ❌ / TikTok ❌）
- 8 条关键假设的优先级评估（影响 × 不确定性矩阵）
- 技术架构建议和 MVP 建议（专注于 YouTube + B站）

**适合谁看**：想知道"为什么做这个、为什么不做那个"的产品决策参考。

---

### 2. `competitor-analysis.md` — 竞品分析

**作用**：摸清市场格局，找到差异化定位

**内容**：
- BibiGPT（100万+用户）和 NoteGPT（40万+安装）的深度分析
- 功能矩阵对比表（10+ 功能维度）
- 定价策略对比（$0 → $99/月）
- **核心发现**：两个竞品都只做"摘要"不做"下载"，这是我们最大的差异化空间
- 本产品的定位建议和风险分析（竞品快速跟进）

**适合谁看**：想知道"市场和竞品什么情况、我们怎么差异化"的战略参考。

---

### 3. `prd.md` — 产品需求文档

**作用**：定义产品做什么、做到什么程度

**内容**（9 个章节）：
1. **Executive Summary** — 一句话定位 + 核心场景
2. **Background & Context** — 问题空间、技术发现、竞品格局
3. **Objectives & Success Metrics** — 3 个目标、5 个非目标、5 个可测试指标（含测试前提和降级判定）
4. **Target Users** — 一级用户（知识型学习者）和二级用户（视频收藏者）
5. **User Stories** — P0 核心 5 条 / P1 次要 3 条 / P2 后续 4 条，每条有条件和验收标准
6. **Solution Overview** — 系统架构、设计决策（含 PO token 和 cookies 方案）、用户流程、AI 摘要格式、界面布局
7. **安全、隐私与运营** — 数据安全策略、版权合规、运营限流和成本预估（日 ¥3）
8. **Open Questions** — 3 个待定问题（部署方案、cookies 自动化、域名）
9. **Timeline & Phasing** — MVP 4-6 周，Phase 2 / Phase 3 规划
    - 附录 A：技术选型原则（为什么选这个，而非版本号）
    - 附录 B：关键风险及缓解方案

**适合谁看**：所有人。这是项目的"宪法"文档，定义做什么和不做什么。

---

### 4. `pre-mortem.md` — 预检分析

**作用**：假设产品已经上线失败了，回头看"为什么会死"，提前防御

**内容**：
- **🔴 致命级×3**：YouTube 反爬、B站 cookies 过期、DeepSeek 限流
- **🟡 严重级×3**：下载太慢、导图拥挤、非技术用户不会用
- **🟢 轻微级×3**：AI 质量波动、粘贴乱链、侵权风险
- 每个风险都有缓解方案和应对策略
- **核心行动项**：从第一天就把"降级方案"写进代码

**适合谁看**：开发前必须读一遍。每个"致命级"风险都要在架构设计中有对应措施。

---

### 5. `user-stories-dev.md` — 开发任务拆解

**作用**：把 PRD 拆成可执行的开发任务，给研发直接参考

**内容**：
- **6 个开发批次**，按依赖链排列，每批产出可测试的功能点
- **18 个具体任务**，每个包含：
  - 涉及的文件路径
  - API 接口定义和请求/响应格式
  - 核心代码骨架（Python 和 Vue）
  - 错误处理策略
- 依赖关系总图（可视化任务先后顺序）
- 技术栈列表（后端 FastAPI / 前端 Vue 3 / markmap / SSE）

**适合谁看**：开发人员。这是从 PRD 到代码的桥梁文档。
